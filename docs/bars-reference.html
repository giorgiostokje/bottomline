Bottomline Bar & Segment Reference Machine-readable documentation for the Bottomline Claude Code status-line plugin

Design Philosophy

Bottomline is a Claude Code status-line plugin. Every time Claude refreshes, bottomline.sh receives a JSON payload on stdin describing the session, renders ANSI output to stdout, and exits. There is no daemon and no persistent state.

The design follows three principles:

  • Passive and fast. The script reads signals that already exist on disk—lockfiles, manifests, config files, binaries on PATH. It never modifies anything, never installs tools, and does not block the terminal.
  • Canonical ordering. Every language bar follows the same six-slot order: Runtime → Package manager → Framework → Add-ons → Testing → Tooling. This makes bars predictable—an LLM or human can find information in the same slot regardless of which language bar is displayed.
  • Layered config. Defaults live in settings.json. A user file at ~/.claude/bottomline.json overrides per key. A project file at <project>/.claude/bottomline.json overrides again. Objects deep-merge; arrays and scalars use the highest-priority non-null value.

Rendering Pipeline

bottomline.sh executes in two phases per refresh:

  1. Main status line. Calls seg() once per built-in segment, then flush() to render them with a gradient background. The gradient is expanded to exactly N stops via piecewise linear RGB interpolation in awk, so gradient endpoints always land on the first and last segment.
  2. Bars. Iterates CFG_BARS (a JSON array). Each entry is a script bar (runs a bars/*.sh file in a subshell) or an inline segment bar (rendered from JSON without a subshell). Auto-detection via auto_bars.scripts in settings.json can prepend bars when signal files are found in the project root.

Gradient interpolation

Given a gradient config of K hex keyframes and N segments, each segment i gets color at position t = i / (N−1) along the keyframe sequence. A solid background is ["#hex"] (K=1). The two-stop form ["#dark","#lighter"] is the most common.

flush() output per segment

bg(i)  SPACE  BOLD  content  bg(i)  SPACE
fg(i)  bg(i+1)  SEP                    ← between segments
RESET  fg(i)   SEP  RESET              ← after last segment

The separator character is U+E0B4 (, rounded powerline right chevron) by default, configurable via segments.separator.

Bar Anatomy

A bar is an additional line rendered below the main status line. It appears when signal files are found in the project root (auto-detection) or when explicitly listed in bars config.

Script bar structure

Every built-in bar in bars/ follows this pattern:

#!/usr/bin/env bash
# 1. Guard — exit early if signal file absent
PROJ="${BOTTOMLINE_PROJECT_DIR:-}"
[[ -z "$PROJ" || ! -f "$PROJ/go.mod" ]] && exit 0

# 2. Source shared helpers (seg, flush, ANSI primitives)
source "$BOTTOMLINE_LIB/helpers.sh"

# 3. Palette — bar-specific colors, bypassed if BOTTOMLINE_BAR_COLORS is set
if [[ -z "${BOTTOMLINE_BAR_COLORS:-}" ]]; then
  FG_TEXT=$(make_fg "$(hex_to_rgb "#c8e8f4")")
  FG_ACCENT=$(make_fg "$(hex_to_rgb "#29bcd8")")
  _bar_gradient='["#031824","#054860"]'
else
  _bar_gradient="$BOTTOMLINE_GRADIENT"   # inherit from main config
fi

# 4. Icons — three branches: nerd / emoji / none
case "$BOTTOMLINE_ICON_TYPE" in
  nerd)  IC_LANG=$'\xee\x9c\xa4' ;;   # raw UTF-8 Nerd Font bytes
  emoji) IC_LANG='🐹' ;;
  *)     IC_LANG='' ;;
esac

# 5. Detection — read manifests, grep deps, check binaries
go_version=$(awk '/^go /{print $2; exit}' "$PROJ/go.mod")

# 6. Segments — add_seg() accumulates; canonical slot order
add_seg "${FG_ACCENT}${IC_LANG} ${FG_TEXT}Go ${FG_ACCENT}v${go_version}"

# 7. Flush — renders with gradient; exits silently if nothing added
(( ${#_sc[@]} == 0 )) && exit 0
flush "$_bar_gradient"

Inline segment bar

Alternatively, a bar can be defined purely in JSON (no script file). The main script renders it using its own seg()/flush() engine:

{
  "bars": [{
    "segments": [{
      "icon": "model",
      "content": "My custom label",
      "colors": { "accent": "#00ff88" }
    }],
    "colors": { "background": "#1a1a2e" }
  }]
}

Canonical Segment Slot Order

Every language ecosystem bar follows this six-slot order. Slots with nothing to show are silently skipped; the order must never deviate.

Detection Signal Priority

Detection runs in this priority order. The first positive signal wins:

  1. Lockfile / manifest dependency — e.g. Cargo.lock, poetry.lock, composer.lock, package.json dep key.
  2. Config file presence — e.g. .golangci.yml, pytest.ini, .rubocop.yml, phpstan.neon.
  3. Binary on PATH — e.g. command -v golangci-lint, command -v shellcheck.

All three signals are valid. Many tools are detectable via more than one.

random-facts bars/random-facts.sh

A utility bar showing a rotating random fact. Not a language bar — no signal file; must be explicitly configured. Colors are fully supplied by the bar config (no built-in palette). Demonstrates custom bars, refresh intervals, and caching.

This bar calls the uselessfacts.jsph.pl API and caches the result in /tmp for the configured refresh_minutes interval (default 60). When offline or the API times out, it selects from a built-in list of 10 facts.
Config keyEffect
refresh_minutesCache bucket duration. A new fact is fetched each time the bucket changes.
colorsRequired — bar has no built-in palette. Accepts any color object or "inherit".

Example configuration

{
  "bars": [{
    "script": "random-facts",
    "refresh_minutes": 30,
    "colors": {
      "text":    "#e2d5c3",
      "accent":  "#da7756",
      "background": ["#2e1f14", "#160f0a"]
    }
  }]
}
